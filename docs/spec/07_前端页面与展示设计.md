# 前端页面与展示设计

## 1. 设计目标

- 一眼看懂流程
- 让流程“动起来”
- 强化人工审批价值
- 突出失败回流亮点
- 支撑比赛演示节奏

## 2. 整体原则

- 信息分层展示
- 核心状态优先展示
- 文档查看与流程查看分离
- 少页面、强表达
- 统一视觉语言

## 3. 页面总览

比赛版固定 4 个核心页面：
1. 需求输入页
2. 流程看板页
3. 审批页
4. 产物页

## 4. 页面一：需求输入页

### 页面目标
- 输入原始需求
- 创建项目
- 启动流程

### 建议结构
- 系统标题
- 项目名称输入框
- 项目描述输入框（可选）
- 原始需求大文本框
- 示例需求区
- `启动流程` 按钮
- 可选 `填充示例需求` 按钮

## 5. 页面二：流程看板页

这是比赛版最核心页面。

### 顶部摘要区
- 项目名称
- 当前状态
- 当前阶段
- 当前版本
- 最近更新时间

### 快速统计卡片
- 任务总数
- 已完成任务数
- 待审批数
- 异常/返修数

### 当前焦点提示条
告诉评委当前系统在做什么，例如：
- 当前正在执行：登录模块返修
- 当前等待人工操作：技术方案审批

### 主流程时间线
建议展示 9 个节点：
1. 需求澄清
2. PRD 生成
3. PRD 审批
4. 技术方案
5. 技术方案审批
6. 开发执行
7. 测试验证
8. 最终评审
9. 交付完成

### 节点状态
每个节点至少支持：
- 未开始
- 进行中
- 已完成
- 已驳回/失败

### 任务面板
重点展示：
- 登录模块：测试失败 → 返修中 → 重测通过
- Todo 模块：已通过

### Agent 状态面板
展示：
- Agent 名称
- 当前状态
- 当前任务
- 最近摘要

### 最近事件流
按时间倒序展示最近 10~20 条关键事件。

## 6. 页面三：审批页

### 目标
- 完成 PRD、技术方案、最终交付的人工审批
- 体现强门禁与回流机制

### 结构
左侧：审批列表
- 审批类型
- 当前状态
- 创建时间
- 是否待处理

右侧：审批详情
- 审批类型
- 所属阶段
- 状态
- 来源任务
- 文档预览
- 审批意见输入框
- `通过` / `驳回` 按钮

### 关键要求
审批不是假按钮，审批后必须：
- 真实改变审批状态
- 真实触发流程推进或回流
- 刷新看板页状态

## 7. 页面四：产物页

### 目标
- 展示全过程文档产物
- 体现“流程不仅在跑，而且真的产出了东西”

### 建议结构
左侧：分类树或标签
- 需求文档
- PRD
- 技术方案
- 任务清单
- 开发产物
- 测试报告
- 交付说明

右侧：内容预览区
- 产物名称
- 类型
- 版本
- 来源任务
- 正文内容

### 导出按钮
建议放在产物页右上角：
- `导出交付包`

## 8. 关键组件建议

- `ProjectSummaryCard`
- `StageTimeline`
- `TaskListPanel`
- `AgentStatusPanel`
- `EventFeed`
- `ApprovalList`
- `ApprovalDetail`
- `ArtifactExplorer`

## 9. 推荐看板布局

### 顶部
- 项目标题
- 当前状态
- 当前阶段
- 快速统计卡片

### 中间
- 主流程时间线

### 下方左侧
- 任务列表

### 下方右侧
- 最近事件流

### 底部或侧边
- Agent 状态卡片

## 10. 比赛演示路径

1. 需求输入页：输入需求并启动
2. 流程看板页：展示需求澄清和 PRD 生成
3. 审批页：PRD 审批
4. 看板页：展示技术方案生成
5. 审批页：技术方案审批
6. 看板页：展示开发/测试与失败回流
7. 产物页：展示测试报告、缺陷说明和交付说明
8. 导出交付包

## 11. 刷新与异常提示

- 看板页 3~5 秒轮询刷新
- 审批提交后立即刷新
- 导出成功后刷新产物列表
- 必须有加载中、提交中、导出中提示
- 必须有失败提示：创建失败、审批失败、文档读取失败、导出失败
