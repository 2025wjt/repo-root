# 开发计划与任务拆分

## 1. 总目标

比赛版开发的核心目标不是“功能越多越好”，而是：

1. 先跑通完整主链路
2. 再补可视化和审批
3. 再打磨失败回流和导出
4. 最后做演示稳定性优化

## 2. 开发原则

- 先闭环，后美化
- 先协议，后编码
- 先主链路，后分支
- 先 Mock 稳定，再接真实 LLM
- 前后端并行，但数据契约先行

## 3. 两人分工建议

### 角色 A：流程底座负责人
负责：
- FastAPI 后端
- SQLite 表结构与数据读写
- Orchestrator
- 状态机落地
- 审批与回流逻辑
- zip 导出逻辑
- 后端 API

### 角色 B：展示与智能负责人
负责：
- React 前端
- 需求输入页、看板页、审批页、产物页
- 文档预览组件
- Agent 提示词/模板
- Clarify / PM / Architect 输出模板
- 演示路径打磨

## 4. 共同冻结项

两人必须一起确认：
- 项目状态枚举
- 任务状态枚举
- 审批状态枚举
- Agent 输入输出结构
- API 请求响应格式
- 数据库存储字段
- 项目目录结构
- 产物命名规则

## 5. 阶段划分

### 阶段一：协议冻结阶段
完成：
- MVP 范围
- 架构
- 状态机
- Agent 协议
- 数据模型
- API 设计

### 阶段二：后端主链路打通阶段
完成：
- FastAPI 初始化
- SQLite 初始化
- 创建项目
- Orchestrator 主链路
- 基础产物落盘
- 后端可通过脚本跑完整 happy path

### 阶段三：前端核心页面打通阶段
完成：
- 需求输入页
- 流程看板页
- 审批页
- 产物页
- 接入项目、任务、审批、产物 API

### 阶段四：失败回流与导出阶段
完成：
- 登录模块测试失败
- 返修任务创建
- 重测通过
- 导出 zip
- 最终评审

### 阶段五：演示打磨与稳定性阶段
完成：
- demo 数据固化
- 页面文案统一
- 异常提示补齐
- 演示路径打磨
- Mock 兜底方案

## 6. 最小任务清单

### 后端
- 初始化 FastAPI 项目
- 建表脚本
- Pydantic 模型
- Repository / Service
- 创建项目逻辑
- 审批提交逻辑
- Orchestrator 主链路
- 回流逻辑
- 导出逻辑

### 前端
- 初始化 React + Vite
- API 请求封装
- 需求输入页
- 看板页
- 审批页
- 产物页
- 任务列表组件
- 时间线组件
- 文档预览组件

### Agent 模板
- 结构化需求模板
- PRD 模板
- 技术方案模板
- 任务拆分模板
- 登录模块测试失败模板
- 登录模块返修模板

## 7. 最推荐优先级

### 第一优先级
- 后端主链路
- 两次审批
- 看板页
- 产物页

### 第二优先级
- 失败回流
- 导出 zip

### 第三优先级
- Agent 状态面板
- 更细的任务详情
- 页面美化

## 8. 最终检查清单

比赛前逐条确认：
- 是否能创建项目
- 是否能看到主流程时间线
- 是否能进行 PRD 审批
- 是否能进行技术方案审批
- 是否能看到开发/测试任务
- 是否能演示一次测试失败回流
- 是否能进行最终评审
- 是否能查看关键产物
- 是否能导出 zip
- 是否有稳定演示脚本
