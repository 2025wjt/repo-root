# 系统总体架构设计

## 1. 架构目标

比赛版总体架构围绕以下四个目标展开：

1. **流程闭环**：支持从需求输入到最终交付的完整链路推进。
2. **编排清晰**：系统需要有统一的流程控制核心，确保多个 Agent 的工作在同一套规则下运行。
3. **状态可见**：项目状态、任务状态、审批状态、Agent 状态和产物状态都应可追踪、可展示、可审计。
4. **演示稳定**：比赛版架构优先保证本地可运行、流程可复现、关键节点可控。

## 2. 核心原则

- 流程优先于智能
- 控制信息与业务产物分离
- Orchestrator 统一推进流程
- 单机统一运行时优先
- 可扩展但不过度设计

## 3. 五层架构

```text
用户交互层
  ├─ 需求输入页
  ├─ 流程看板页
  ├─ 审批页
  └─ 产物页

流程编排层
  ├─ Pipeline Definition
  ├─ Orchestrator
  └─ Policy Engine

Agent 执行层
  ├─ Clarify Agent
  ├─ PM Agent
  ├─ Architect Agent
  ├─ Dev Agent
  └─ Test Agent

状态与产物层
  ├─ SQLite
  ├─ 文件系统
  └─ 项目目录

基础运行层
  ├─ FastAPI
  ├─ Pydantic
  └─ 本地运行时
```

## 4. 模块职责

### 4.1 用户交互层
负责系统与人之间的交互：
- 输入原始需求
- 展示流程状态
- 执行审批操作
- 查看产物内容

### 4.2 Pipeline Definition
定义流程骨架：
- 阶段划分
- 节点顺序
- 输入输出
- 准入准出条件
- 是否需要人工审批
- 失败处理策略

### 4.3 Orchestrator
运行时中枢，负责：
- 创建项目实例
- 创建任务
- 分配执行 Agent
- 接收执行结果
- 推进状态
- 创建审批记录
- 处理失败回流与暂停

### 4.4 Policy Engine
用于定义：
- 审批规则
- 失败回流规则
- 局部返修规则
- 项目暂停规则
- 最终评审规则

### 4.5 Agent 执行层
负责生成业务内容：
- Clarify：结构化需求
- PM：PRD
- Architect：技术方案与任务清单
- Dev：模块实现与返修
- Test：测试验证与缺陷说明

### 4.6 状态与产物层
- SQLite：保存项目、任务、审批、事件、产物索引、Agent 状态
- 文件系统：保存所有文档正文和导出包

## 5. 核心数据流

### 5.1 项目启动流
1. 用户输入需求
2. 创建项目
3. 初始化状态与首个任务
4. Clarify Agent 输出结构化需求

### 5.2 文档生成流
1. PM Agent 输出 PRD
2. 人工审批 PRD
3. Architect Agent 输出技术方案和任务清单
4. 人工审批技术方案

### 5.3 开发测试流
1. Orchestrator 创建开发/测试任务
2. Dev Agent 执行开发
3. Test Agent 执行测试
4. 失败时生成缺陷并触发返修

### 5.4 最终交付流
1. Human Reviewer 最终评审
2. Orchestrator 更新项目完成态
3. 生成交付说明并导出 zip

## 6. 为什么采用单机统一运行时

1. 降低实现复杂度
2. 提高演示稳定性
3. 降低联调成本
4. 便于未来演进到服务化
