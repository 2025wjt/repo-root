# 比赛版 MVP 范围冻结

## 1. 项目一句话定义

本项目是一个 **AI 驱动的需求交付流程引擎 Demo 平台**。

平台接收用户输入的原始需求后，通过统一的 Pipeline 与 Orchestrator，驱动多个 Agent 按照固定流程完成：

**需求澄清 → PRD 生成 → 技术方案 → 人工审批 → 开发任务派发 → 测试验证 → 评审确认 → 最终交付**

并在整个过程中提供：

- 流程状态可视化
- Agent 执行状态展示
- 关键节点人工审批
- 失败回流与返修机制
- 最终交付物沉淀

## 2. 比赛版核心目标

比赛版 MVP 不是为了证明“AI 写代码特别强”，而是为了证明以下三点：

1. **全链路成立**：系统能够把原本割裂的需求交付链路串成一个完整闭环。
2. **编排能力成立**：系统不是简单地堆叠多个 Agent，而是通过统一流程引擎进行编排和推进。
3. **治理能力成立**：系统能够在关键节点引入人工审批，并支持失败处理、状态追踪、返修回流与结果沉淀。

## 3. 演示需求

固定使用一个简单业务需求作为演示案例：

### 推荐演示需求
开发一个简单的 **Todo Web App**，要求支持：

- 用户登录
- 待办事项新增
- 待办事项删除
- 待办事项修改
- 待办事项完成/未完成状态切换

## 4. MVP 必须实现的能力

### 4.1 需求输入
- 用户通过页面输入自然语言需求
- 创建新的项目实例
- 保存原始需求

### 4.2 需求澄清与结构化
- Clarify Agent 输出结构化需求单
- 产物：`structured_requirement.md/json`

### 4.3 PRD 自动生成
- PM Agent 基于结构化需求生成简版 PRD
- 产物：`prd.md`

### 4.4 第一个人工审批节点：PRD 审批
- 支持通过 / 驳回 / 填写意见
- 驳回后回到 PRD 修改环节

### 4.5 技术方案与任务拆分
- Architect Agent 输出技术方案、接口说明、任务清单
- 产物：`architecture_design.md`、`task_list.json`

### 4.6 第二个人工审批节点：技术方案审批
- 支持通过 / 驳回 / 填写意见
- 驳回后回到技术方案修改环节

### 4.7 开发任务派发
至少创建：
- 2 个开发任务
- 2 个测试任务

推荐拆分：
- 模块 A：用户登录
- 模块 B：待办 CRUD / 状态切换

### 4.8 测试验证
- Test Agent 输出测试报告
- 支持通过/失败结论

### 4.9 至少一次失败回流
推荐演示逻辑：
1. 登录模块首次开发完成
2. 测试失败
3. 生成缺陷说明
4. 回流给 Dev Agent 返修
5. 返修后重测通过

### 4.10 评审确认节点
- 测试通过后不能直接交付
- 需进入最终人工评审

### 4.11 最终交付
最少交付物包括：
- 结构化需求单
- PRD
- 技术方案
- 任务清单
- 测试报告
- 最终交付说明

## 5. MVP 不做什么

- 不做真实分布式部署
- 不做复杂多租户权限系统
- 不做企业级消息中间件
- 不做复杂 RAG
- 不追求强真实代码生成
- 不做完整 DevOps 平台
- 不做无限自治通用 Agent

## 6. 页面范围

比赛版前端固定为 4 个核心页面：

1. 需求输入页
2. 流程看板页
3. 审批页
4. 产物页

## 7. 完成标准

当以下条件全部满足时，视为 MVP 完成：

- 能从需求输入跑到最终交付
- 中间包含至少 2 个审批节点
- 包含至少 1 次失败回流
- 最终项目状态可进入完成态
- 至少具备 4 个核心页面
- 至少生成 6 类交付文档
- 能稳定演示 1 次完整流程
