# 2026-04-20 用户问题与改进诉求

## 背景

用户在体验 `dashboard`、`approvals`、`artifacts` 页面时，针对任务辨识、审批可读性、产物命名、导出时机和导出结果表达提出了一组连续问题。

这些问题集中反映出一个核心缺点：

- 当系统中存在多个任务、多个阶段、多个审批记录和多个产物时，当前界面无法让用户快速判断“当前看到的内容究竟属于哪个阶段、哪个任务实例、哪个导出结果”。

## 用户提出的主要问题

### 1. 任务与阶段辨识度不足

- 当存在多个任务并行或连续执行时，页面上无法清楚区分当前审批记录、当前产物、当前导出结果分别对应哪个任务。
- 用户无法直观看出某条审批记录属于哪个阶段，也无法知道它对应的是哪个具体任务实例。
- 仅显示 `prd_review`、`architecture_review`、`final_review` 这类英文或技术型标识，不利于业务视角理解。

### 2. 审批页表达不清晰

- 审批页面当前更像“审批记录列表”，而不是“按阶段和任务实例组织的审批工作台”。
- 用户指出：无法分辨这是执行哪个任务的审批页面。
- 审批页中的标题、阶段名、目标文件名，缺少统一的业务语义表达。

### 3. 产物页表达不清晰

- 产物列表中，当前文件名和摘要不足以帮助用户快速判断该文件属于哪个阶段、哪个任务、哪个模块。
- 当执行多个任务后，用户无法判断导出出来的是哪个任务对应的程序或交付结果。
- 当前导出成功提示只显示一条相对路径，对普通用户不够直观。

### 4. 导出时机不合理

- 用户明确指出：在项目未彻底完成之前，应该没有最终交付物，不应允许导出。
- 当前系统允许在未完成完整流程前触发导出，这会让用户误以为导出结果已经具备完整交付意义。
- 这会模糊“阶段性文档产物”和“最终交付物”之间的边界。

### 5. 导出结果语义不清

- 用户进一步追问当前导出的 zip 是“可执行程序代码”还是“半成品”。
- 实际检查后，当前导出 zip 更接近阶段性文档归档，而不是一个可直接运行的程序交付包。
- 页面文案没有明确区分“文档归档包”和“可运行交付包”。

### 6. 文件命名不统一

- 用户希望同一个阶段的文件采用规范化命名。
- 用户明确提出：最好使用中文的 `.md` 文件名称。
- 当前混合存在英文文件名、技术型文件名、缩写型文件名，不利于非技术用户理解。

## 用户已明确表达的偏好

### 1. 页面主标识方式

针对“如何区分多个任务”的问题，用户明确选择：

- 使用 `阶段 + 任务实例` 的双标识方式。

这意味着后续审批页、产物页、导出信息页都应优先采用类似如下表达：

- `PRD 阶段 / PRD 生成任务`
- `技术方案阶段 / 技术方案生成任务`
- `开发阶段 / 登录模块开发任务`
- `测试阶段 / 登录模块测试任务`
- `最终评审阶段 / 最终评审任务`

### 2. 文件命名偏好

用户明确希望：

- 文件名称具备阶段规范性
- 同阶段文件命名统一
- 尽量使用中文 `.md` 文件名

## 当前问题的本质总结

当前系统更偏向“内部实现视角”展示数据，而不是“用户任务视角”展示流程。

具体表现为：

- 记录存在，但语义弱
- 文件存在，但归属弱
- 审批存在，但上下文弱
- 导出存在，但完成态约束弱

因此当任务数量增加、阶段增多、审批链路拉长之后，用户会迅速失去定位能力。

## 后续改进方向建议

### 1. 审批页

- 每条审批记录展示：阶段名称、任务实例名称、审批对象名称、审批状态
- 使用中文业务名称替代内部英文类型名
- 在详情区明确展示：
  - 所属阶段
  - 所属任务
  - 审批对象
  - 关联文件
  - 当前状态

### 2. 产物页

- 每个产物展示：
  - 所属阶段
  - 所属任务实例
  - 文件类型
  - 文件版本
- 导出成功提示应展示更明确的说明，而不是仅给相对路径

### 3. 导出控制

- 未完成最终评审前，不应允许最终交付导出
- 必须明确区分：
  - 阶段性文档产物
  - 最终交付包
- 最终交付包的触发条件应与项目完成态绑定

### 4. 文件命名

- 统一采用面向业务理解的中文命名规则
- 同阶段的 Markdown 文件名称保持一致的前缀和格式

## 待后续补充确认

以下点尚未最终确认，需要在后续设计中继续细化：

- 未完成时的导出按钮是隐藏、置灰，还是允许点击后提示原因
- 中文文件名的完整命名规范
- 最终导出包是否需要真正包含可运行源码，而不只是文档归档
